1. Identifying the Most Expensive Function
While `buyTicket` and `commitHash` have the highest raw maximum gas costs
due to expensive storage writing operations (SSTORE), the function
specifically targeted for optimization is `revealAndDraw`. It is a
computationally heavy function containing complex keccak256 hashing,
blockhash lookups, and multiple state reads.

2. Before and After Gas Numbers
Based on the Foundry gas-report logs, the metrics for the `revealAndDraw`
function show a distinct reduction in gas consumption after applying the

- Before Optimization (lottery.sol): 
  Min: 24,517 | Avg: 44,121 | Median: 53,788 | Max: 53,788

- After Optimization (gasOptimisedLottery.sol): 
  Min: 24,517 | Avg: 44,031 | Median: 53,635 | Max: 53,635

Total Savings: The optimizations successfully reduced the maximum execution
cost by 153 gas and the average cost by 90 gas per call.

3. Why Optimization Was Needed
In the Ethereum Virtual Machine (EVM), interacting with state variables
stored on the blockchain (SLOAD operations) is highly expensive.

- A "cold" read (first access in a transaction) costs 2,100 gas.
- A "warm" read (subsequent access in the same transaction) costs 100 gas.
- Reading a variable from local memory or the stack costs roughly ~3 gas.

Additionally, Solidity 0.8.0+ introduces built-in safety checks for math
operations, including a zero-divisor check (ISZERO, PUSH, JUMPI opcodes)
that executes before every division or modulo operation, adding unnecessary
gas overhead if the divisor is already known to be non-zero.

4. How It Was Optimized
The gas reduction of 153 gas was achieved by combining two distinct
optimization strategies:

A. Removing Redundant Math Checks (`unchecked` block)
In the original contract, calculating the winner index used a standard
modulo operation (% totalParticipants). Because Solidity cannot look across
different functions to prove that totalParticipants is greater than zero
at this exact moment, it injects a safety check to prevent dividing by zero.
However, the contract's phase architecture guarantees totalParticipants > 0
because the closeSale() function explicitly requires it before the lottery
can ever reach the Committed or Drawn phases.

* The Fix: By wrapping the modulo operation in an `unchecked { }` block,
  we explicitly instruct the compiler to drop the zero-divisor check,
  saving the overhead of those opcodes.

B. Explicit Memory Caching (Preventing Stack Shuffling Penalties)
When complex block scopes (like the unchecked block) are introduced, the
Solidity compiler's optimizer can lose track of storage pointers, forcing
it to fall back to expensive "warm" SLOAD reads and triggering
stack-shuffling operations (SWAP, DUP).

* The Fix: To secure the gas savings from the unchecked block, explicit
  memory caching was implemented. State variables were read from storage
  exactly once, saved to local memory, and referenced locally thereafter:
  
  - Caching targetBlock: Stored as _targetBlock to validate require
    statements and fetch the futureBlockHash.
  
  - Caching winner: Stored locally as _winner to be passed into the
    WinnerDrawn event without re-reading from storage.
  
  - Caching prizePool: Read into _prizePool prior to the external call
    and event emission.
