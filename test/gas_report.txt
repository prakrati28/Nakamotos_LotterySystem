1. Identifying the Most Expensive Function
While `buyTicket` and `commitHash` have the highest raw maximum gas costs 
due to expensive storage writing operations (SSTORE), the function 
specifically targeted for optimization in these files is `revealAndDraw`. 
It is a computationally heavy function containing complex keccak256 
hashing and multiple state reads.

3. Why was Optimization Needed
In the Ethereum Virtual Machine (EVM), interacting with state variables 
stored on the blockchain (SLOAD operations) is highly expensive. 

- The first time a state variable is read in a transaction (a "cold" 
  read), it costs 2,100 gas. 
- Subsequent reads of that exact same variable in the same transaction 
  (a "warm" read) cost 100 gas each. 
- In contrast, reading a variable from local memory or the stack costs 
  roughly ~3 gas. 

The unoptimized lottery.sol contract forces the EVM to perform multiple 
redundant "warm" reads by calling round.targetBlock, round.winner, and 
round.prizePool directly from storage multiple times.

4. How It Was Optimized 
The gasOptimisedLottery.sol file fixes this inefficiency by implementing 
a memory caching strategy. Instead of reading from the `round` storage 
pointer repeatedly, the contract reads the state variable once, saves it 
to a local memory variable, and references that local variable thereafter:

- Caching targetBlock: The code stores round.targetBlock into a local 
  variable (_targetBlock). It uses this cheap stack variable for the two 
  require statements and to fetch the futureBlockHash, skipping multiple 
  SLOADs.

- Caching winner: Instead of writing the winner to storage and then 
  reading it back out for the event emission, it caches it locally as 
  _winner, writes it to storage, and passes _winner to the event.

- Caching prizePool: The code reads the prize pool into a local variable 
  (_prizePool) right before the transfer and event emission, preventing 
  the need to fetch it from storage during the WinnerDrawn event.
